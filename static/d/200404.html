Apr.12,2004 (Mon)

■spam処理の話 [COMP]

+ 今までHIMIKO(local)側だけでbsfilterを動かしてspamフィルタしていたのだが、AKANE(メールサーバ)にsshで入ってメールを読み書きすることもあるし、やはりAKANE側でspamフィルタをした方がよいだろうということで、ちょいと手を打ってみた。

+ bsfilterのimapを直接叩くのは単純でよいが、メール到着のタイミングと振り分けのタイミングが一致しないのが気にくわない。bsfilterやPOPFileのようなpop proxy機能も楽だが、pop3以外に応用が効かないので却下。postfix内でbsfilterを通しsieveで振り分けるのがcyrus-imapd的にはきれいだが、bsfilterのデータベースをユーザー毎に分ける簡単な方法が思いつかなかったので断念。AMaViS-newからSpamAssasinを呼ぶ手もあるけどSpamAssassinについて知らなすぎるので断念 :-)

+ で、結局.forward経由でpostfixからcourier maildropを呼んで、maildrop内でbsfilterを通してcyrus deliverに食わせるようにしてみることにした。で、以下のように適当に.forwardと.mailfilterをでっちあげてみたところ特に問題は発生しない様子(*1)。

 "| exec /usr/local/bin/maildrop || exit 75"
DELIVER="| /usr/local/cyrus/bin/deliver -a $LOGNAME"
USERID="$LOGNAME"
xfilter "bsfilter --pipe --insert-flag --insert-probability --auto-update"
if (/^X-Spam-Flag: Yes/)
{
  to "$DELIVER -m spam振り分け先 $USERID"
}
to "$DELIVER $USERID"

+ AKANE上のEmacs+Wanderlustから読む場合は以上で終わり。HIMIKOのMeadow+WanderlustからAKANEのbsfilterに学習させるために以下の設定をHIMIKOの.wlに追加する。

(setq elmo-spam-bsfilter-shell-program "c:/Program Files/PuTTY/plink.exe")
(setq elmo-spam-bsfilter-shell-switch "")
(setq elmo-spam-bsfilter-program "")
(setq elmo-spam-bsfilter-args '("-l" "remoteユーザ名" "imapサーバ名" "bsfilter"))

+ 普通sshとssh-agentを使うところだが、WindowsなのでここはPuTTYのplinkを使ってみた。事前にPagentの設定をしておかないとエラーになるので注意。

+ とりあえず、こんな感じでしばらく様子を見てみよう。

*1: でも、この辺書き間違えると周囲に迷惑をかけたりできるのでcut&pasteで使おうなんて思わないこと。

Apr.25,2004 (Sun)

■その後のHIMIKO [COMP]

+ CPUクーラーのファンを7→8cmダウンバースト+Panafloファンにして、電源を鎌力にしたら静かになったのだが、CPU温度が常時51,2℃で負荷時に64℃とかでどうにもならない。電源もやけに熱くなって夏場が怖い。

+ ケース横をちょっとあけるとけっこう温度が下がったので、底部ファンだけでは吸気が足りないっぽい。ふと思いついて後部ケースファンを排気から吸気に変更したら温度が5,6℃下がって電源もぬるくなった。ぐっど。

+ CPU直上に外気が入ることで、CPUと電源に行く空気の温度が下がったのが勝因だろうか。まぁ、しばらくこれで様子を見ていこう。だめならまたケース交換だ。

■その後のAKANE [COMP]

+ Pentium4-2.53GHzからPentium4-3.06GHzに交換。クロックが2割向上+HyperThreadingはどの程度の効果かなと思ったけど、悪くないな。いろいろと反応がよくなった予感。

+ こっちも問題はCPU温度だなー。Pentium4は温度に関してAthlon64より厳しいからより深刻かもしれない。配置上、下部にファンがある電源にしてHIMIKO同様CPU付近で吸気を強化すればよさそうだ。あ
